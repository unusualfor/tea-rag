# tea-rag

A queryable explorer for Francesco's tea collection. Browse, search, and ask questions about ~23 teas from Japan, China, India, and beyond.

**Live at [tea.francescoforesta.com](https://tea.francescoforesta.com)**

## What this is

A personal project to catalog and explore a growing tea collection. Two modes: **Browse** (grid with filters and vector search) and **Ask** (conversational AI that can search the collection, look up brewing parameters, and recommend teas). Built entirely on Cloudflare's free tier — Workers, Vectorize, R2, and Workers AI — as a learning project for RAG patterns.

## How it works

```
┌─────────────────────────────────────────────────────────┐
│                     Browser (React SPA)                 │
│                                                         │
│   Browse mode              Ask mode                     │
│   ┌─────────────┐         ┌──────────────────┐          │
│   │ Grid + Filters│        │ Chat interface   │          │
│   │ Vector search │        │ Inline tea cards │          │
│   └──────┬──────┘         └────────┬─────────┘          │
└──────────┼─────────────────────────┼────────────────────┘
           │                         │
           ▼                         ▼
┌──────────────────────────────────────────────────────────┐
│                Cloudflare Worker (Hono)                   │
│                                                          │
│  GET /api/teas        POST /api/chat                     │
│  POST /api/search     (tool-calling loop)                │
│       │                    │          │                   │
│       ▼                    ▼          ▼                   │
│  ┌──────────┐    ┌──────────────┐  ┌──────┐              │
│  │ Vectorize │    │  Workers AI  │  │  R2  │              │
│  │ (768-dim) │    │ Llama 3.3   │  │photos│              │
│  │ BGE-base  │    │ 70B fp8     │  │      │              │
│  └──────────┘    └──────────────┘  └──────┘              │
└──────────────────────────────────────────────────────────┘
```

**Browse mode**: The frontend fetches all teas from `/api/teas`. Filters (category, urgency, country) narrow the list client-side. The search bar sends queries to `/api/search`, which generates an embedding via Workers AI and queries Vectorize for semantic matches.

**Ask mode**: User messages go to `/api/chat`, which runs a tool-calling loop with Llama 3.3 70B. The model has four tools — `search_teas`, `list_teas_by_filter`, `get_tea`, and `list_categories` — to query the collection. Tool results feed back into the model, which produces a streaming text response with inline `[tea:id]` references rendered as clickable cards.

## Tech stack

- **TypeScript** — end to end
- **React 18** + **Vite** — client-side SPA
- **shadcn/ui** (v4, base-ui) + **Tailwind CSS** — component library and styling
- **Hono** — lightweight Worker API framework
- **Cloudflare Workers** — edge runtime
- **Cloudflare Vectorize** — vector database (768-dim, cosine similarity)
- **Cloudflare Workers AI** — embeddings (`bge-base-en-v1.5`) and chat (`llama-3.3-70b-instruct-fp8-fast`)
- **Cloudflare R2** — photo storage
- **YAML files** — source of truth for tea data

## Local development

```bash
# Clone and install
git clone https://github.com/francescoforesta/tea-rag.git
cd tea-rag
npm install

# Parse all tea YAML files and generate src/data/teas.json
npm run ingest

# Start Vite dev server (frontend on :5173, proxies /api/* to :8787)
npm run dev

# In a separate terminal, start the Worker dev server
# --experimental-vectorize-bind-to-prod connects to the live Vectorize index
npx wrangler dev --experimental-vectorize-bind-to-prod
```

The Vite dev server proxies `/api/*` to the Worker on port 8787. Workers AI and Vectorize connect to remote Cloudflare resources; R2 is simulated locally.

## Adding and managing teas

### Add a new tea

1. Create `content/teas/{id}.yaml` using an existing file as a template.
2. Run `npm run ingest` to validate and regenerate `src/data/teas.json`.
3. Optionally upload a photo:
   ```bash
   npx wrangler r2 object put tea-photos/photos/{id}.jpg --file ./path/to/photo.jpg
   ```
4. Deploy: `npm run deploy`.
5. Push embeddings to Vectorize by calling the ingest endpoint:
   ```bash
   curl -X POST https://tea.francescoforesta.com/api/ingest \
     -H "Authorization: Bearer $INGEST_SECRET" \
     -H "Content-Type: application/json" \
     -d @dist/ingest-payload.json
   ```

The ingest handler detects new, changed, and unchanged teas via content hashing. Only new or modified teas are re-embedded.

### Update a tea

Edit the YAML file, run `npm run ingest`, deploy, and re-ingest. The content hash detects the change and re-embeds only the modified tea.

### Remove a tea

1. Delete the YAML file from `content/teas/`.
2. Run `npm run ingest` to regenerate data.
3. Vectors for deleted teas remain in Vectorize but won't match any tea in the dataset — they're effectively orphaned and harmless.
4. Commit and deploy.

## Cost and quotas

Everything runs on Cloudflare's free tier:

- **Workers AI**: daily neuron quota (free tier). For a personal site with light traffic, this is more than enough. If you hit the limit, chat responses will fail for the day — browse mode is unaffected.
- **Vectorize**: free tier covers the small index (23 vectors, 768 dimensions).
- **R2**: free tier covers photo storage at this scale (5 GB storage, 10 GB egress/month).
- **Workers**: 100,000 requests/day on free tier.

No external API keys (Anthropic, OpenAI, etc.) are needed.

## Rate limiting

The `/api/chat` endpoint has an in-memory rate limiter: **5 requests per minute per IP**. This resets on Worker restart and is deliberately simple for the expected traffic.

## Design decisions

See [DECISIONS.md](DECISIONS.md) for the full decision log. Highlights:

- **All-Cloudflare stack**: no external LLM providers — Workers AI keeps everything on one platform with zero API key management.
- **Two-pass tool loop**: Llama 3.3 70B tends to keep calling tools indefinitely. The chat handler caps at 2 tool rounds, then forces a text response.
- **Four tools, not more**: `search_teas`, `list_teas_by_filter`, `get_tea`, `list_categories`. Clear separation helps the model choose correctly.
- **Bracket syntax for references**: `[tea:id]` in LLM output → inline clickable cards in the UI.
- **YAML as source of truth**: diffs are readable, no database to manage for ~40 items.
